Technical Part 5 05/08/17

    Tell your non-tech friend a story about a time you:

    1. Got blocked on a simple problem
    2. Solved a problem in an elegant way

    In both cases:

  • What was the problem?
    • On this particular webpage there were 9 dots on the screen, and when you clicked on them they would change a different colour depending on how you clicked on them. There was also a counter at the side that would tell you how many of that colour there were at any given time. My problem was figuring out a way to scan all the dots, check their colour, then add it to the counter.

  • What problem solving techniques did you use?
    • I first went to the Slack Channel to see if anyone else was having a similar problem. I also had Sachin look over it in person, but many of these terms were new to both of us. I got talking to Daz on the Slack Channel and she pointed out that I had missed a bracket and had spelled something wrong. I tried again but ran into new errors. Copying these errors into Google, solved one issue but another would pop up. I started to look at certain lines individually and printing them to the ‘console’, this helped me see what each part was doing. Finally I could see a clear path and got it to work!

  • How did you feel throughout the process?
    • I felt frustrated but also determined to get it working. When I got it to work though I was over the moon.

  • What did you learn?
    • I learnt to look thoroughly at my spelling and syntax. Also to ask for help, another pair of eyes can spot something you might not. I also found the debugger a great tool, only after I'd done this exercise.

    Reflect on how confident you feel using the problem solving techniques and process:

  • Pseudocode
    • I really benefit from using pseudocode, as it helps me break down the process of what I've got to do and also for when I run into problems.

  • Trying something
    • I definitely benefited from trying things and search on Google for how to approach the concept.

  • Rubber ducky method
    • This is good, I've used it a couple of times, I also like to write down what I know/don't know and what I need it to do.

  • Reading error messages
    • In the end, reading the error messages helped me nail the problem, knowing what line to focus on and using Google to determine what the error meant.

  • Console.logging
    • I was able to see what information was coming out of certain fuctions and loops by adding the console.log, I'll be sure to use it again.

  • Googling
    • People on the net seemed to have had similar issues, just in different contexts, so being able to break down what they were trying to do and what I would like to do helped me to know which parts I need to use in my code and how that code works.

  • Asking your peers for help
    • Another great life-saver! I was able to get a second opinion on my code and things like tiny spelling mistakes that you can overlook may be picked up on.

  • Asking coaches for help
    • Another great thing about Slack, I had my question on JavaScript answered by a coach and that was very helpful.

  • Improving your process with reflection
    • Any time I make a mistake, I write it down and add it to my notes like "don't forget to use this" or "a value you must be added here".

    Explain to your non-tech friend Using the JS Array docs as a starting point, describe what these functions do in your own words:

  • .map()
    • Will create a new array from an existing one that applies a function to all the elements inside that array without affecting the original.

  • .filter()
    • Creates a new array from an existing one but only lets through certain information depending on what condition you give it.

  • .reduce()
    • Reduces an array to a single value by applying a function against an accumulator.
